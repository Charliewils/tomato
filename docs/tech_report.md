# 基于边缘AI的番茄温室智能监测系统——技术报告

> **项目类型**：学科竞赛作品 | **团队**：三人 | **周期**：2026年4月–7月
> **定位**：RV1126B端侧离线接地式番茄病虫害决策助手，NPU检测+LLM/CPU语音异构调度

---

## 一、项目概述

本系统以嵌入式AI开发板ELF-RV1126B为核心，集成计算机视觉、大语言模型、语音交互和多源传感器，构建一套**完全离线可运行的温室番茄智能监测系统**。系统具备以下核心能力：

1. **实时病虫害检测**：YOLOv8n 13类目标检测，覆盖9种常见番茄病害+3级果实成熟度，板端推理 ~25 FPS
2. **产量动态估算与预测**：基于检测框面积的MLP校准模型 + 时序预测器，输出当前估产和未来7天可采收量
3. **接地式语音问答**：离线LLM注入实时传感器数据与作物记录，支持自然语言查询温室内状态
4. **环境多维感知**：三路I2C传感器实时采集光照、土壤湿度、CO₂浓度、温湿度
5. **数字孪生可视化**：React Three Fiber构建3D温室看板，手机扫码即达

系统强调**端侧离线、异构调度、接地真实**——所有AI推理均在板端NPU/CPU完成，无需云端即可闭环运行，同时保留云端升级链路。

### 2.2 I2C 传感器总线

板子提供多路I2C，传感器经PCA9548A多路复用器（后简化为直连）分挂不同总线：

```
i2c-3 (21120000.i2c) —— ADS1115 ADC @0x48 —— 电容式土壤湿度探头
i2c-4 (21130000.i2c) —— VEML7700 光照 @0x10
                    —— SCD41 CO₂/温湿度 @0x62
```

传感器数据由sensors.py统一采集，每5秒写入`/userdata/sensors.json`，通过systemd服务`crop-sensors.service`常驻运行。

## 三、传感器系统

### 3.1 环境光传感器 VEML7700

| 参数 | 规格 |
|------|------|
| 接口 | I2C，地址0x10，挂载i2c-4 |
| 量程 | 0–120k lux（自动量程切换，5档增益×积分时间） |
| 分辨率 | 0.0036 lux/count（最高灵敏档） |
| 读取方式 | ioctl I2C_RDWR repeated-start 读ALS寄存器 |
| 数据字段 | `lux`, `lux_raw`, `sim_veml` |

设计要点：VEML7700读取ALS寄存器必须使用repeated-start时序（I2C_RDWR ioctl 0x0707双msg），分离的write+read事务会读到全零数据。配置更改后需等待2.5×积分时间使旧数据排出。自动量程算法从低灵敏档向高灵敏档扫描，饱和时回退上一档。板子重启后偶有ENXIO掉线，已做try/except优雅降级mock兜底。

### 3.2 土壤湿度传感器（电容式探头+ADS1115 ADC）

| 参数 | 规格 |
|------|------|
| 探头类型 | 电容式土壤湿度探头（模拟电压输出） |
| ADC | ADS1115，16位ΔΣ，I2C地址0x48，挂载i2c-3 |
| ADC配置 | ±4.096V满量程，128SPS，单次转换模式，AIN0-GND |
| 标定 | 干端(空气) 2.772V → 0%，湿端(泡水) 1.540V → 100% |
| 数据字段 | `soil_v`(原始电压), `soil`(湿度%), `sim_soil` |

湿度计算公式：

$$\text{soil\%} = \frac{2.772 - V_{soil}}{2.772 - 1.540} \times 100\%$$

ADS1115通过读取config寄存器（0x01）验证在位状态，不在位时`available=False`自动退mock。电容式探头输出电压与介电常数成反比——干土介电常数低、电压高；湿土介电常数高、电压低。

### 3.3 CO₂/温湿度传感器 SCD41

| 参数 | 规格 |
|------|------|
| 接口 | I2C，地址0x62，挂载i2c-4 |
| 测量原理 | 光声NDIR（CO₂）+ 片上CMOS（温湿度） |
| CO₂精度 | ±40 ppm + 5%读数 |
| 读取方式 | stop → start_periodic → data_ready → read（9字节，含3路CRC8校验） |
| CRC8多项式 | 0x31，初始值0xFF |
| 数据字段 | `co2`(ppm), `temp`(℃), `rh`(%), `sim_scd` |

SCD41不响应i2cdetect的quick-write扫描（i2cdetect看不到0x62属正常），通过实际发送stop命令+捕获OSError判定是否在位。周期性测量模式5秒一次，`read_all`中预留最多6秒等待data_ready。CRC8校验失败抛出IOError，上层捕获后降级mock。

### 3.4 统一数据输出

三路传感器由`sensors.py --daemon`统一采集，输出JSON格式到`/userdata/sensors.json`：

```json
{
  "ts": 1783164331,
  "lux": 1045.1,    "lux_raw": 567,    "sim_veml": false,
  "soil_v": 2.70,    "soil": 5.8,       "sim_soil": false,
  "co2": 715,        "temp": 25.0,      "rh": 65.7,    "sim_scd": false
}
```

`sim_*`字段标记数据来源：`false`为真实传感器，`true`为mock兜底（正弦波模拟，维持系统不崩溃）。任一传感器读失败不影响其他通道。

|------|------|

### 4.2 检测类别（13类）

| 类别 | 类型 | 说明 |
|------|------|------|
| bacterial_spot | 叶部病害 | 细菌性斑点病 |
| early_blight | 叶部病害 | 早疫病 |
| late_blight | 叶部病害 | 晚疫病 |
| leaf_mold | 叶部病害 | 叶霉病 |
| leaf_miner | 虫害 | 潜叶蝇 |
| mosaic_virus | 病毒病 | 花叶病毒 |
| septoria | 叶部病害 | 斑枯病 |
| spider_mites | 虫害 | 红蜘蛛 |
| yellow_leaf_curl_virus | 病毒病 | 黄化曲叶病毒 |
| healthy | — | 健康叶片（检测但不显示） |
| green | 果实成熟度 | 青果 |
| half_ripened | 果实成熟度 | 半熟果 |
| fully_ripened | 果实成熟度 | 完熟果（可采收） |

### 4.3 推理性能

- **NPU池**：rknnpool模块管理8个RKNN实例，NPU时分复用
- **帧率**：稳定 24–26 FPS（同时运行检测+NPU LLM）
- **与LLM共存验证**：后台压NPU检测30秒694帧无崩，LLM成功加载作答（仅自身被争用拖慢至21秒）
- **OBJ_THRESH**：0.25（综合）/ 可切换模式改变过滤策略

### 4.4 检测模式

触屏按钮循环切换三种模式：
- **综合**：病害+果实全检
- **仅果实**：过滤病叶类，乘远摄补正系数×1.25
- **仅病叶**：过滤果实类，跳过产量CSV日志防污染历史

## 五、产量估算与预测

### 5.1 产量校准器

采用训练的MLP替代固定均重法：

| 组件 | 参数 |
|------|------|
| YieldCalibratorCPU | 291参数，7.4g MAE，R²=0.88 |
| YieldPredictorCPU | 545参数，R²=0.59 |
| 推理框架 | 纯numpy，无框架依赖 |
| 输入特征 | 24维时序特征（各类果实检测框像素面积×平滑计数） |
| 外推防护 | ±3σ截断 |

### 5.2 7日产量预测

- **数据源**：当日滑动窗口均值 + 前3天日均检测数据（按天聚合）+ 温度(sensors.json)
- **预测输出**：未来7天预计可采收量（克），写入CSV `f7d_g`列
- **冷启动保护**：历史<2天显示"数据积累中"，防稀疏数据外推误导
- **更新频率**：30秒

### 6.3 接地式Prompt设计

系统通过`crop_context()`动态构建上下文，注入实时数据：

```
【环境】温度24.1℃，湿度66%，CO₂ 715 ppm，光照1045 lux
【选株】8株逐株：第1株青3半熟2…第5株未记录…
```

### 6.4 云端LLM升级（可选）

系统支持云端LLM自动兜底——优先走智谱GLM-4-Flash（免费，延迟~3s，回答质量显著优于0.5B），失败自动退本地RKLLM。架构为通用OpenAI兼容接口，支持多后端failover。

## 七、语音交互系统

### 7.1 整体架构

```
用户语音 → VAD录音 → SenseVoice ASR → LLM(云端优先/本地兜底)
                                                    ↓
用户听到 ← aplay播放 ← Piper TTS(分句流式) ← 回答文本
                         ↑
                   Edge TTS云端(优先) → 失败退Piper
```

### 7.2 自动语音识别（ASR）

| 组件 | 规格 |
|------|------|
| 框架 | sherpa-onnx 1.13.2 |
| 模型 | SenseVoice Small（int8量化，229 MB） |
| 语言 | 中文，ITN后处理 |
| 实时率(RTF) | 0.39（CPU 4线程） |

### 7.3 语音活动检测（VAD）

采用silero_vad.onnx（628KB），替代固定时长盲录：
- 等待说话才开始录音（最长等10秒）
- 静音0.6秒自动判定结束
- 最长录12秒防无限等待
- 动态选麦：按设备名rank自动选择USB麦

### 7.4 语音合成（TTS）

| 方案 | 模型 | RTF(相机满载) | 音质 |
|------|------|:---:|------|
| 首选 | Edge TTS（微软云端，zh-CN-XiaoxiaoNeural） | — | 自然 |
| 板端 | Piper huayan-medium (63 MB) | 1.18 | 清晰自然 |
| 弃用 | Fanchen-C (121 MB) | 8.0 | 同理但太慢 |
| 弃用 | eSpeak-ng | 0.012 | 机械音中文听不清 |

**关键发现**：板载codec I2S配置为双声道，喂单声道数据→2倍速播放（音调升八度）。通过在`/etc/asound.conf`预设`crop_spk`虚拟设备（plug自动单→双声道上混）修复。此bug曾误导TTS选型一整周。

### 7.5 分句流式播放

Piper逐句合成→临时wav入队列→播放线程异步消费，首句合成完即开播，后续句合成与播放流水线重叠，首声延迟大幅度缩短。

## 十一、关键设计决策与踩坑记录

### 11.1 数据驱动 vs 模拟

系统全链路真实数据优先，传感器缺失/故障自动mock兜底（`sim_*`字段标记），确保系统不崩溃。但植株/抓拍/产量数据不造假——板端未检测到果实时显示"待记录"而非填充假数据。

### 11.2 NPU异构调度

YOLOv8检测与RKLLM共存于同一NPU。实测验证时分复用可行：检测维持23+ FPS，LLM仅自身被争用拖慢，不拖累检测。

### 11.3 音频2倍速Bug

板载I2S配置双声道，单声道数据按双声道时钟2倍速播放→音调升八度。此bug被误判为TTS模型问题，绕了espeak/mbrola/fanchen/Piper一整圈。最终修复仅一行asound.conf。教训：硬件音频异常先计时验物理播放率。

### 11.4 PlantVillage数据集陷阱

v1模型使用PlantVillage单叶灰底数据集训练，真实田间检测完全失效——该数据集标注≈整图大框（只学分类没学定位），域分裂严重。这是领域公认陷阱（Frontiers综述fpls.2024.1493322）。v4改用田间整株数据集，检测质量从全废跃升至静态图55张100%命中。

### 11.5 0.5B小模型Prompt反噬

曾给LLM prompt加"每句话简短"指令，0.5B反而写结构化长清单撞token上限截断。去掉该指令恢复原prompt后答案反而短而完整。小模型对句式约束类指令容易过度发挥。

## 十二、技术创新点

1. **全端侧离线闭环**：13类视觉检测 + LLM推理 + 语音交互 + 传感器融合，全部在RV1126B单板完成，不依赖互联网
2. **单NPU时分复用**：YOLOv8（8实例池）与Qwen2.5-0.5B共享同一NPU，验证工业可行性
3. **接地式LLM**：实时传感器+逐株记录+室外天气注入prompt，LLM输出具体数据引用的农业建议
4. **MLP产量校准**：训练的291参数模型替代固定均值，逐类果实框面积→物理重量，R²=0.88
5. **数字孪生数据驱动**：3D温室场景受真实光照/土壤数据实时驱动视觉，逐株标签同步板端记录
6. **异构TTS链**：云端Edge TTS → 本地Piper自动降级；分句流式播放使首声延迟最小化

## 十四、总结

本系统围绕"端侧AI+接地真实"的核心设计理念，在RV1126B嵌入式平台上实现了视觉检测、大语言模型、语音交互和传感器感知的异构融合。系统可完全离线运行，同时保留云端升级链路作为增强选项。项目适用于温室精准农业场景，为学科竞赛提供了完整的演示闭环：打开电源→自动检测→自然语音问答→手机扫码查看3D看板。
